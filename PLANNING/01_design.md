## 2. Entry schema

```yaml
# entries/01-frameworks/tradingagents.yaml
slug: tradingagents                       # required, unique, kebab-case, matches filename
title: TradingAgents                      # required, official project/paper title
type: framework                           # required, enum: framework | paper | dataset | benchmark | tool | environment | list
category: 01-frameworks                   # required, matches directory
subcategory: multi-agent                  # optional, free-form tag for sub-section ordering
url: https://github.com/TauricResearch/TradingAgents   # required, primary canonical URL
paper_url: https://arxiv.org/abs/2412.20138            # optional, when an entry has both
authors:                                  # required for papers; optional for repos
  - Yijia Xiao
  - Edward Sun
  - Di Luo
  - Wei Wang
maintainer: Tauric Research                # optional, for repos
year: 2024                                 # required for papers (year of first arXiv submission)
venue: arXiv (q-fin.TR)                    # optional but recommended for papers — venue or "arXiv preprint"
metrics:                                   # optional, all keys optional, all labelled with as_of date
  stars: 73700
  citations: null                          # integer when known
  as_of: "2026-05"
description: >-                           # required, one neutral sentence — what it is, not what it claims
  Multi-agent LLM trading framework with specialised analyst, researcher, trader, and
  risk-manager roles, coordinated via LangGraph and supporting multiple LLM backends.
note: >-                                  # required, the honest annotation (1–3 sentences)
  The paper (arXiv 2412.20138, v7) reports superior cumulative returns, Sharpe, and
  max drawdown vs. baselines on a stock-trading task. As with any 2024–25 LLM trading
  study, readers should independently check whether the evaluation window post-dates
  the model's training cutoff, and whether transaction costs are modelled.
tags:                                     # optional, free-form
  - multi-agent
  - langgraph
  - stocks
verification:
  status: verified                        # enum: verified | pending | dropped
  sources:                                # at minimum the canonical URL + one corroborating source
    - https://arxiv.org/abs/2412.20138
    - https://github.com/TauricResearch/TradingAgents
  verified_on: "2026-05-14"
  notes: null                             # populated only on drop/pending
```

**Required fields:** `slug`, `title`, `type`, `category`, `url`, `description`, `note`, `verification.status`, `verification.sources` (≥ 1), `verification.verified_on`.

**Schema enforcement:** `validate_entries.py` rejects any entry missing a required field, with an unknown `type` or `category`, or with a `verification.status == verified` but fewer than one source. It also enforces global slug uniqueness and dedupes by URL.

**What the note may NOT contain:**
- Speculation about authors' intent or competence.
- Performance numbers we cannot trace to a source URL listed in `verification.sources`.
- Marketing language ("revolutionary", "state-of-the-art", "groundbreaking").
- Recommendations to buy/sell/trade anything.

## 4. Tooling design

### 4.1 `tools/linkcheck.py`
- Reads every `entries/**/*.yaml`, extracts `url` + `paper_url` + every URL in `verification.sources`.
- `HEAD` request first; falls back to `GET` if the server doesn't support HEAD (arXiv does, GitHub does, most don't have an issue).
- 5-second connect timeout, 15-second read timeout, single retry on transient 5xx.
- Outputs a Markdown table to stdout: `URL | status | latency-ms | entry-slug`.
- Exit code 0 if all `verified` entries' URLs return 2xx/3xx; exit 1 otherwise.
- CI runs it weekly (Mon 09:00 UTC) and on every PR that touches `entries/`.

### 4.2 `tools/validate_entries.py`
- Loads `entries/**/*.yaml` via PyYAML safe-load.
- Validates against the schema in §2 (Python-defined, no jsonschema dep needed for ~25-field flat schema).
- Checks: required fields present; `type` ∈ enum; `category` matches directory; slug uniqueness across all categories; URL uniqueness for `verified` entries (warning, not error, since some entries legitimately share a maintainer URL); `verification.status == verified` ⇒ `verification.sources` non-empty.
- Lints the note for forbidden marketing words (`revolutionary`, `state-of-the-art`, `groundbreaking`, `breakthrough`, `cutting-edge`) — emits a warning, not an error, so honest use in a quote is still possible.
- Exit code 0 on clean, 1 on any error.

### 4.3 `tools/build_readme.py`
- Reads entries, groups by `category`, sorts within category by (`subcategory`, `-metrics.stars` for repos / `-year` for papers, `title`).
- Renders a single `README.md` from a template file (`tools/README.template.md`) with placeholders for each section.
- Deterministic: same inputs → same output bytes (sorted keys, fixed line endings).
- Run via `python tools/build_readme.py --check` in CI to assert the committed README matches the entries.

### 4.4 CI workflows

`.github/workflows/validate.yml` — runs on every PR:
1. `pip install -r tools/requirements.txt`
2. `python tools/validate_entries.py`
3. `python tools/build_readme.py --check`

`.github/workflows/linkcheck.yml` — runs weekly + on PR touching `entries/`:
1. Same setup.
2. `python tools/linkcheck.py`
3. On weekly failure, opens an issue tagged `stale-link`.

## 5. Contribution flow

**`CONTRIBUTING.md`** requires that every new-entry PR:

1. Adds a single YAML file under the correct `entries/<category>/` directory.
2. Fills every required schema field, including ≥ 1 `verification.sources` URL.
3. Includes the honest note conforming to the rules in `00_think.md` §4 (factual, fair, non-promotional).
4. Passes `validate.yml` and `linkcheck.yml` CI.
5. Does not re-add an entry recorded as `dropped` in `PLANNING/02_verification_log.md` unless the PR explains what materially changed.

**`PULL_REQUEST_TEMPLATE.md`** is a checklist:

```
- [ ] One entry per PR (or a clearly-scoped batch with rationale)
- [ ] Verification source(s) attached
- [ ] Honest note is factual, fair, and non-promotional
- [ ] Not financial advice in any text added
- [ ] `python tools/validate_entries.py` passes locally
- [ ] `python tools/build_readme.py` was re-run and README is up to date
```

**Issue templates:** structured forms for "suggest an entry" and "report a stale or incorrect entry" — both pre-fill the verification-source field so suggestions arrive review-ready.
